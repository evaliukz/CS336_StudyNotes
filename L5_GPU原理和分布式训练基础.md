# 一、这章到底在讲什么

你可以把这一章理解成一句话：

现在训练和推理大模型，瓶颈很多时候已经不是“会不会算”，而是“数据搬得够不够快”。
所以这章不是在教你背 CUDA 语法，而是在教你一个更重要的思维方式：

GPU 优化 = 想办法少搬数据、把数据放近一点、让一群线程整齐地干活。

这也是讲义里反复在强调的核心：GPU 的计算能力，特别是矩阵乘法能力，增长得比内存带宽更快，所以你不能只盯着 FLOPs，还得盯着 memory hierarchy（内存层级） 和 data movement（数据搬运）。  ￼

⸻

# 二、先建立最重要的直觉：CPU 和 GPU 到底差在哪

### 1）CPU 追求“单个任务快完成”

CPU 更像一个很聪明的办公室白领：
它擅长处理复杂逻辑、分支判断、各种杂事，目标是让单条任务延迟低。所以 CPU 会放很多资源在控制、缓存、分支预测这些地方。  ￼

### 2）GPU 追求“总共干更多活”

GPU 更像一个超大工厂：
它不追求某一条线程飞快结束，而是追求同时让海量线程一起干活，把总吞吐量拉高。GPU 因此会放很多很多算术单元，弱化复杂控制。  ￼

所以一句大白话总结：

CPU = 少量聪明工人，适合复杂活。
GPU = 海量普通工人，适合整齐划一的大批量活。  ￼

⸻

# 三、GPU 里面最关键的几个东西

### 1）SM、SP、Thread、Warp、Block 是什么

这块第一次看很容易晕，我给你翻译成人话。

Thread

最小干活单位。
你可以把它理解成“一个小工人”。  ￼

Warp

32 个线程组成一个 warp。
GPU 调度时，很多时候不是一个线程一个线程调，而是按 warp 这个小队来调。也就是说，这 32 个线程倾向于一起执行同一条指令。  ￼

Block

一组线程组成一个 block。
一个 block 会被放到某个 SM 上跑，而且 block 内的线程可以共享 shared memory。  ￼

SM（Streaming Multiprocessor）

SM 可以理解成 GPU 里的“车间”。
一个 SM 能同时承载一个或多个 block，在里面调度很多 warp 执行。  ￼

SP / CUDA core

你可以把它理解成更底层的算术执行单元，是真正在做加减乘除这些操作的地方。  ￼

⸻

### 2）SIMT 到底是什么

这一章会提到 SIMT: Single Instruction, Multiple Threads。
大白话就是：

同一波线程，做同样的动作，只是处理不同的数据。

比如 32 个线程一起做 y[i] = relu(x[i])，代码逻辑一样，只是每个线程处理的 i 不同。  ￼

为什么这很重要？
因为 GPU 最喜欢这种“大家排好队、动作整齐一致”的活。只要线程们开始有人往左走、有人往右走，效率就会掉。  ￼

⸻

# 四、GPU 的内存层级：这章最值钱的地方之一

这章反复强调：离计算单元越近的内存越快，但越贵、越小。  ￼

你可以这样记：

### 1）Register

线程私有，最快，最小。
像工人手里直接拿着的工具。  ￼

### 2）Shared Memory / L1

在 SM 里面，很快，block 内线程能共享。
像车间里的工作台，工人们可以反复拿来用。  ￼

### 3）L2 Cache

整个芯片范围共享，比 shared memory 慢。
像工厂内部仓库。  ￼

### 4）Global Memory（HBM/显存）

容量大，但慢得多。
像工厂外面的大仓库，拿货最慢。  ￼

所以这一章的优化哲学其实很朴素：

少去大仓库拿东西，多在手边和车间里复用。  ￼

⸻

# 五、为什么 GPU 会慢：Roofline 这个脑图一定要建立起来

<img width="1993" height="1201" alt="image" src="https://github.com/user-attachments/assets/6651e90e-88f8-4365-948c-422819a8f2e4" />


这一章会用 Roofline model 去解释性能瓶颈。
你不用记公式，先记结论：

一个算子慢，通常就两种原因：
	1.	算力不够，计算单元满了
	2.	带宽不够，数据搬运跟不上  ￼

这里最关键的词叫 arithmetic intensity（计算强度），意思是：

每搬 1 字节数据，你做了多少计算。

如果你每次搬很多数据，但只做一点点算术，那就是memory-bound（访存瓶颈）。
如果你数据搬得不多，但算得非常猛，那才可能是compute-bound（计算瓶颈）。  ￼

对于很多现代深度学习 workload，这章强调的判断是：

大部分时候更容易先卡在内存带宽，而不是算力。
所以优化优先级通常是：先想怎么减少访存，再想怎么堆计算。  ￼

# 六、这章最核心的 6 个优化技巧

下面这 6 个你最好真的记住，因为后面看 CUDA、Triton、FlashAttention、torch.compile，基本都绕不开它们。公开讲义镜像和笔记都把这些作为本讲核心优化点。  ￼

⸻

### 技巧 0：控制流分歧（control divergence）

它是什么

同一个 warp 里的线程，本来应该整齐执行同一条指令。
但如果你写了 if/else，结果 32 个线程里一半走 if，一半走 else，GPU 往往要两边都跑一遍，只是每次只让部分线程生效。  ￼

为什么慢

因为本来 32 个人能一起走，现在变成：
   •   先让一半人干 A
   •   再让另一半人干 B

等于队伍不整齐了。  ￼

大白话记忆

GPU 怕大家意见不统一。
warp 最喜欢“32 个人做同一件事”。  ￼

⸻

### 技巧 1：低精度计算（low precision）

核心直觉

位数越低，数据越小，搬起来越轻。
比如 FP16 比 FP32 占的字节少，所以同样的数据通路能塞更多值，算子更容易变快。  ￼

讲义里举的 ReLU 例子，FP32 需要搬更多字节，FP16 搬的数据更少，所以 arithmetic intensity 更高。  ￼

但为什么不会精度崩掉

因为现代 GPU 的 Tensor Core 往往会做低精度输入、较高精度累加。
也就是：搬运和部分计算用低精度省流量，关键累加阶段保一些精度。  ￼

大白话记忆

低精度不是只为了“省显存”，更是为了“少搬数据、提吞吐”。  ￼

⸻

### 技巧 2：算子融合（operator fusion）

它是什么

把多个本来分开的 kernel 合成一个。
因为如果你每做一步都把中间结果写回 global memory，再读出来给下一个算子，那就会来回折腾显存。  ￼

经典直觉

讲义镜像里举了 sin^2 + cos^2 的例子：
如果拆开做，可能要发很多个 kernel；如果融合起来，一次 kernel 就能把整条链做完。  ￼

为什么快

因为你把“来回运货”变少了。
中间结果尽量留在寄存器或片上，而不是反复进出显存。  ￼

大白话记忆

融合不是为了代码好看，是为了别总把半成品送回大仓库。  ￼

⸻

### 技巧 3：重计算（recomputation）

它是什么

有些中间结果，别存。
后面需要的时候，重新算一遍。  ￼

为什么这听起来反直觉

因为直觉上你会想：
“都算过了，为什么不存下来，非要重算？”

但 GPU 上很多时候 算一下比存一下、再读一下更便宜。
因为算力很强，但显存访问很贵。  ￼

讲义镜像里用 stacked sigmoid / 反向传播举例，强调重计算可以减少 memory read/write 次数。社区笔记里也提到一个示意例子：前向+反向从 8 次访存降到 5 次。  ￼

大白话记忆

GPU 世界里，记住东西不一定比重新想一遍便宜。  ￼

⸻

### 技巧 4：内存合并访问（memory coalescing）

<img width="2132" height="1143" alt="image" src="https://github.com/user-attachments/assets/04c81011-ca76-479d-b47d-d8e791631cc4" />

它是什么

GPU 访问 DRAM / HBM (global memory) 时，查一个数值，会返回一整段连续的数值。
如果一个 warp 里的线程访问的是连续地址，那这些访问就更容易合并成少量大块请求。还是为了矩阵乘法。

为什么重要

因为内存本身常常是按 burst mode（突发方式）工作的。
你要是访问得很零碎，GPU 就得发更多请求，效率就差。  ￼

大白话例子

最好情况：
32 个线程刚好拿一排连续的数据。

糟糕情况：
32 个线程东一榔头西一棒子，各拿各的零碎地址。

前者像快递一次装满一车，后者像一单一单地跑。  ￼

大白话记忆

线程访问地址要排队站好，别乱。  ￼

⸻

### 技巧 5：分块（tiling）

<img width="2015" height="1157" alt="image" src="https://github.com/user-attachments/assets/4c23c3f4-9de3-497a-b88a-0a6f916cb867" />

还是利用了矩阵乘法的性质，将矩阵分割成子矩阵，再相乘计算。

这是这章最重要的“大招”

Tiling 的思想是：

不要每次都从 global memory 反复拿同样的数据。先把一小块数据搬到 shared memory，再反复用。  ￼

为什么矩阵乘法特别适合 tiling

矩阵乘法会反复用到相同的输入块。
如果你不分块，那很多元素会被重复从全局内存读很多次。
如果你分块，先把 tile 放到 shared memory，就能在 SM 内部反复复用。  ￼

这本质上做了什么

把：
   •   慢的全局重复读取
变成
   •   快的片上重复读取  ￼

难点是什么

tile size 不是越大越好，因为还要平衡：
   •   shared memory 大小
   •   是否利于 coalescing
   •   矩阵维度能不能整齐切开
   •   寄存器/occupancy 等资源限制  ￼

大白话记忆

Tiling = 先囤一小车货到车间，再在车间里反复加工。  ￼

⸻

# 七、为什么矩阵乘法形状会突然变慢

这一章还会解释一个很“玄学”但其实很工程的现象：

矩阵只大一点点，性能可能突然掉下去。  ￼

社区笔记和讲义镜像都提到一个经典例子：
在 A100 上，某个 tile 设定下，矩阵从 1792 变到 1793，需要的 tile 数会从 98 跳到 120；而 A100 只有 108 个 SM，于是原来一轮能跑完的活，现在得分两轮，性能就突然掉。  ￼

大白话就是：

不是你只多了 1 行数据这么简单，而是调度分块方式变了，GPU 车间突然装不下这一波任务了。

这类现象常被归因到：
   •   对齐问题
   •   tile 切分变化
   •   wave quantization / SM 利用率变化  ￼

⸻

# 九、FlashAttention 用大白话怎么理解

### 1）普通 attention 为什么慢

普通 attention 里，QK^T 会生成一个很大的 attention score 矩阵。
然后你还要：
   •   读它做 softmax
   •   再读它乘 V

也就是说，这个大中间结果会被写到 HBM / global memory，再读回来好几次。
数学没问题，但访存特别惨。  ￼

### 2）FlashAttention 的核心想法

不要把整个大 attention matrix 存下来。

而是：
   •   把 Q、K、V 分块
   •   算一小块，就立刻做后续处理
   •   能不落到 HBM 就不落
   •   最后直接累积出输出  ￼

所以它本质上不是“少算了很多数学”，而是：

同样的数学，换了一种更尊重 GPU 内存层级的执行方式。  ￼

<img width="2048" height="1137" alt="image" src="https://github.com/user-attachments/assets/b71edf70-c597-452e-ac72-f9f1c30c5ae2" />

⸻

### 3）难点：softmax 不是要看完整一行吗

对，这就是最妙的地方。
普通 softmax 你会觉得必须先看到整行，才能知道最大值和分母。社区笔记也把这个问题点出来：标准 softmax 往往至少要多次遍历数据。  ￼

FlashAttention 靠的是 online softmax 思想：
在一块一块处理数据时，持续维护：
   •   当前最大值
   •   当前归一化分母的统计量

这样即使没有看到全量数据，也能边扫边更新，最后得到和标准 softmax 一致的结果。  ￼

你可以这样记

普通 softmax 像：
   •   先把一整本书都搬来
   •   找最大值
   •   再重读一遍算分母
   •   再重读一遍算结果

online softmax 像：
   •   书一页一页送来
   •   我边看边维护“到目前为止的最大值”和“累计统计”
   •   最后不需要把整本书一直摊在桌上  ￼

⸻

### 4）FlashAttention 到底快在什么地方

这讲强调的重点是：

FlashAttention 主要优化的是 IO / memory traffic，不是大幅减少理论计算量。
它把 attention 从“很容易 memory-bound”的实现，变得更接近“把 GPU 算力真正榨出来”的实现。  ￼

大白话总结：

FlashAttention 快，不是因为它更“聪明地偷工减料”，而是因为它不再傻乎乎地反复搬大矩阵。  ￼

⸻

# 十、你真正要记住的“章眼”

这一章如果你只记 6 句话，记这 6 句就够了：

	1.	GPU 追求吞吐，不追求单线程低延迟。  ￼
  
	2.	GPU 的性能经常不是算得不够快，而是数据搬得不够快。 
  ￼
	3.	离 SM 越近的内存越快，所以要尽量把数据留在寄存器/shared memory。  ￼
  
	4.	优化重点：减少访存、合并访存、重复利用片上数据。  ￼
  
	5.	六大招：避免分歧、低精度、融合、重计算、coalescing、tiling。  ￼
  
	6.	FlashAttention 本质上是“用 tiling + online softmax + fusion，避免把大 attention 矩阵反复写回显存”。  ￼

# 十一、给你的复习笔记版

下面这份你可以直接存着背。

CS336 Lecture 5: GPUs 笔记

1. 本章目标
   •   让 CUDA / GPU 没那么“玄学”
   •   理解 GPU 什么时候快，什么时候慢
   •   学会从数据搬运角度理解优化
   •   用这些思想解释 FlashAttention 为什么快  ￼

2. CPU vs GPU
   •   CPU：延迟优化，适合复杂控制流
   •   GPU：吞吐优化，适合大规模并行、规则计算
   •   GPU 通过大量简单计算单元提升总吞吐，而不是单线程速度  ￼

3. GPU 执行模型
   •   Thread：最小工作单位
   •   Warp：32 个线程一起执行
   •   Block：线程组，运行在一个 SM 上，可共享 shared memory
   •   SM：执行 block 的主要硬件单元
   •   SIMT：同一条指令作用于多个线程，不同线程处理不同数据  ￼

4. GPU 内存层级
   •   Register：最快，线程私有
   •   Shared memory / L1：快，SM 内，block 内共享
   •   L2：芯片级共享
   •   Global memory / HBM：最大最慢
   •   原则：越靠近计算越快，但越小  ￼

5. Roofline 思想
   •   性能瓶颈要么是算力，要么是带宽
   •   arithmetic intensity = FLOPs / bytes moved
   •   强度低 → memory-bound
   •   强度高 → 更可能 compute-bound
   •   现代深度学习很多 workload 先受限于带宽  ￼

6. GPU 优化六大招
   •   控制流分歧：warp 内分支不一致会变慢
   •   低精度：减少数据搬运，提高算术强度
   •   算子融合：减少中间结果写回显存
   •   重计算：有时重新算比存取更划算
   •   内存合并访问：warp 访问连续地址更高效
   •   分块 tiling：把数据搬到 shared memory 反复复用  ￼

7. Tiling 的本质
   •   从“重复访问 global memory”
   •   变成“把 tile 放进 shared memory 后重复使用”
   •   是矩阵乘法高性能实现的核心思想之一  ￼

8. 形状为什么影响 matmul 性能
   •   不同矩阵维度会影响 tile 切分、对齐、SM 利用率
   •   某些尺寸只差 1，也可能让 tile 数量激增，导致性能突降  ￼

9. FlashAttention 的核心
   •   标准 attention 会反复把大中间矩阵写回/读出 HBM
   •   FlashAttention 通过分块处理 Q/K/V
   •   用 online softmax 实现 tile-by-tile 的归一化
   •   目标是减少 HBM 访问，提升 IO 效率  ￼

10. 一句话总复习

GPU 优化的核心不是“多写点神秘 CUDA”，而是“少搬数据、把数据放近、让线程整齐、把中间结果尽量留在片上”。  ￼
